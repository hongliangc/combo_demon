- 明确指定输出目标文件

- 生成完整的开发文档

- 包含代码示例和架构图

## 有效提示词模式总结

### 模式 1：任务延续型

**格式**：
```
继续完成 [任务名称]
```

**适用场景**：
- 会话中断后继续
- 上下文已建立
- 任务目标明确

**示例**：
- "继续完成boss多段攻击"
- "继续优化性能"
- "继续修复bug"

---

### 模式 2：列表式需求

**格式**：
```
1. [需求1]
2. [需求2]
3. [需求3]
```

**适用场景**：
- 多个独立任务
- 需要系统化处理
- 任务优先级明确

**优势**：
- 清晰的任务边界
- 便于进度追踪
- AI自动创建TodoList

---

### 模式 3：分层需求描述

**格式**：
```
1. [主要求]
   1.1 [子需求]
   1.2 [子需求]
2. [主要求]
   2.1 [子需求]
```

**适用场景**：
- 复杂任务分解
- 多级需求说明
- 需要详细规范

**优势**：
- 结构化清晰
- 避免遗漏细节
- 便于AI理解深度

---

### 模式 4：框架+细化

**格式**：
```
[总体目标]

核心要点：
- [要点1]
- [要点2]
- [要点3]

可以 [具体方向]
```

**适用场景**：
- 文档生成
- 架构设计
- 代码重构

**示例**：
```
整理Boss战斗系统设计

核心要点：
- 状态机设计
- 连击系统
- 性能优化

可以补充UML图、代码示例、调用链
```

---

### 模式 5：问题+方案验证

**格式**：
```
问题：[具体问题描述]
当前方案：[解决思路]
需要确认：[不确定的点]
```

**适用场景**：
- Debug调试
- 方案选型
- 技术咨询

**优势**：
- 明确问题域
- 展示思考过程
- 获得针对性建议

---

## 提示词质量要素

### 1. 明确性

**好的示例**：
```
修复Hahashin攻击的时候还可以移动的问题
```

**不好的示例**：
```
修复移动问题
```

**差异**：
- 明确了角色（Hahashin）
- 明确了状态（攻击时）
- 明确了问题（还可以移动）

---

### 2. 上下文充分

**好的示例**：
```
在develop_log.md中按时间倒序整理，最新的在上面
```

**不好的示例**：
```
整理文档
```

**差异**：
- 指定了文件
- 指定了排序方式
- 明确了目标格式

---

### 3. 可操作性

**好的示例**：
```
添加UML类图、状态转换图、调用链图
```

**不好的示例**：
```
添加一些图
```

**差异**：
- 具体的图表类型
- 可直接执行
- 结果可验证

---

### 4. 结构化

**好的示例**：
```
1. boss和player都添加血条
2. 修复攻击时移动问题
3. 修复翻滚时移动问题
```

**不好的示例**：
```
添加血条，修复移动
```

**差异**：
- 编号清晰
- 一项一行
- 便于追踪

## 提示词反模式（避免）

### 反模式 1：过于模糊

❌ **错误示例**：
```
优化一下
```

✅ **正确示例**：
```
优化Boss状态机性能，减少不必要的状态切换
```

---

### 反模式 2：多任务混杂

❌ **错误示例**：
```
修复bug，添加功能，写文档
```

✅ **正确示例**：
```
1. 修复攻击时移动的bug
2. 添加血条UI组件
3. 整理设计文档到develop_log.md
```

---

### 反模式 3：缺少上下文

❌ **错误示例**：
```
这个怎么实现
```

✅ **正确示例**：
```
Boss多段连击系统怎么实现：
- 支持配置攻击序列
- 支持时序控制
- 支持参数化
```

---

### 反模式 4：没有验收标准

❌ **错误示例**：
```
实现血条
```

✅ **正确示例**：
```
实现血条系统：
- Boss和Player都有血条
- 支持平滑动画
- Boss血条根据阶段变色
- 显示当前/最大血量
```

---

## 总结：高质量提示词的黄金法则

1. **具体化**：明确对象、状态、期望结果
2. **结构化**：使用列表、层级、编号
3. **上下文**：提供必要的背景信息
4. **可验证**：结果可测试、可检查
5. **分步骤**：复杂任务拆分为小步骤
6. **有约束**：明确质量标准和限制条件
7. **可迭代**：允许渐进式完善
